# Document Processing Service

## Что делает сервис
Сервис хранит метаданные документов и поддерживает:
- Создание документа;
- Получение документа (с историей);
- Поиск/листинг документов;
- Пакетную отправку на утверждение и пакетное утверждение;
- Фоновые воркеры для массовой обработки (submit/approve);
- Регистрация фактов утверждения в таблице `approve_register`.

## Архитектура и технологии
- Java 21, Spring Boot 4
- PostgreSQL
- Liquibase для миграций

- `me.tongfei.progressbar.ProgressBar` для отображения прогресса в консоли

## Конфигурация и запуск
Для конфигурации сервиса обработки документов, вы можете открыть файл `DocumentProcessingService/src/main/resources/application.yml` и настроить следующие параметры:
 - `server.port` — порт, на котором будет работать приложение (по умолчанию 8080).
 - `spring.datasource.url` — URL для подключения к базе данных PostgreSQL.
 - `spring.datasource.username` и `spring.datasource.password` — учетные данные для доступа к базе данных.
 - `workers.batch.size` — количество документов, обрабатываемых за одну итерацию воркера.
 - `workers.submit.*` и `workers.approve.*` — настройки для фоновых воркеров, включая задержку перед первым запуском и интервал между запусками (в миллисекундах).
 - `scheduling.pool.size` — количество потоков для работы воркеров.

### Запуск приложения
Предварительно: убедитесь, что у вас установлены Java 21+, Maven и Docker (если будете использовать Docker).

### 1) Запуск через Maven (локально)
1) Подготовьте PostgreSQL и создайте базу/пользователя согласно `DocumentProcessingService/src/main/resources/application.yml`:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/doc_db
    username: service_account
    password: password
```
2) Подготовьте файл `DocumentProcessingService/src/main/resources/application.yml` с правильными данными (следует заменить переменные для Docker-compose на необходимые параметры по конфигурации, приведенной выше).

3) Сборка и запуск:

```bash
cd DocumentProcessingService
mvn clean package -DskipTests
java -jar target/DocumentProcessingService-1.0.0.jar
```

4) Проверьте, что приложение запустилось и подключилось к базе данных (логи должны показать успешное подключение и запуск).

### 2) Запуск через Docker / docker-compose
В корне репозитория есть `docker-compose.yml`, который поднимает Postgres и приложение.

```bash
docker-compose up --build
```

После запуска проверьте, что приложение запустилось и подключилось к базе данных (логи должны показать успешное подключение и запуск).

## Запуск генератора документов

Для генерации тестовых документов можно использовать `DocGeneratorService`, который представлен в виде отдельного модуля. Он позволяет создать конкретное количество документов, которое описано в конфигурационном файле для этого модуля (`DocGeneratorService/src/main/resources/application.yml`):

```yaml
server:
  port: 8081

generator:
  n: 10000
  url: "http://localhost:8080/api/v1/document-processing/create"
  author-name: "Generator service"
  title-prefix: "Generated Document"
```

В этой конфигурации описываются следующие параметры:
- `server.port` - порт, на котором будет работать генератор (по умолчанию 8081)
- `generator.n` - количество документов, которое будет создано через API сервиса обработки документов
- `generator.url` - URL эндпоинта для создания документа в сервисе обработки документов
- `generator.author-name` - имя автора, которое будет использоваться для всех сгенерированных документов
- `generator.title-prefix` - префикс для заголовков сгенерированных документов (к которому будет добавляться порядковый номер)

Для запуска генератора следует выполнить следующие команды:

```bash
cd DocGeneratorService
mvn clean package
java -jar target/DocGeneratorService-1.0.0.jar
```

В момент запуска генератор начнет отправлять POST-запросы на указанный URL для создания документов. В логах генератора будет отображаться прогресс создания документов, а также статистика по успешным запросам и времени выполнения всей операции создания.

## API (ключевые эндпоинты и DTO)
Базовый префикс: `/api/v1/document-processing`.

1) Создание нового документа
- POST /api/v1/document-processing/create
- Request DTO: `CreateDocumentRequestDto { author, title }`
- Response: `DocumentResponseModelDto`

Пример:
```bash
curl -X POST http://localhost:8080/api/v1/document-processing/create \
  -H 'Content-Type: application/json' \
  -d '{"author":"Alice","title":"My doc"}'
```

2) Получение конкретного документа с возможностью получить историю изменений
- GET /api/v1/document-processing/{documentId}?withHistory=true

3) Получение списка документов по id (для массовой обработки)
- GET /api/v1/document-processing/documents?documentIds=f4qfuowrebfcqsd3&size=10
- Есть поддержка пагинации (size/page), сортировки (sortBy=createdAt:DESC) и фильтрации по статусу (status=APPROVED)

4) Поиск документов с фильтрами (`status`, `author`, `fromDate`, `toDate`)
- GET /api/v1/document-processing/search?status=SUBMITTED&author=John&fromDate=2026-01-01&toDate=2026-01-31
- Есть поддержка пагинации (size/page), сортировки (sortBy=createdAt:DESC) и фильтрации по статусу (status=APPROVED)

5) Отправка пачки документов на утверждение
- POST /api/v1/document-processing/operations/submit

Example:
```bash
curl -X POST http://localhost:8080/api/v1/document-processing/operations/submit \
  -H 'Content-Type: application/json' \
  -d '{"documentIds":["f4qfuowrebfcqsd3"],"approver":"automator","details":"batch submit"}'
```

6) Утверждение пачки документов
- POST /api/v1/document-processing/operations/approve

7) Тест конкурентного утверждения (для проверки обработки конфликтов)
- POST /api/v1/document-processing/operations/test-concurrent-approve

## Воркеры
В проекте два фоновых воркера в `org.itqProj.worker`:

1) `SubmitWorkerImpl` - периодически берет документы в статусе `DRAFT` и инициирует процедуру отправки на утверждение для каждого документа.
2) `ApproveWorkerImpl` - берет документы в статусе `SUBMITTED` и инициирует процедуру утверждения каждого документа.

Поведение воркера
- Берёт список идентификаторов через запрос к базе данных.
- Обрабатывает каждый ID последовательно в рамках одной итерации; считает статистику (success/conflict/not found).
- Логирует результаты и отображает прогресс через прогресс-бар.

Как конфигурировать
- `workers.batch.size` - сколько ID читать за проход.
- `workers.*.initialDelay` - задержка перед первым запуском (мс).
- `workers.*.fixedDelay` - интервал между запусками (мс).
- `scheduling.pool.size` - число потоков планировщика, чтобы запуск нескольких воркеров мог происходить параллельно.

Примечание: прогресс-бар выводится в stdout; в контейнере логи собираются системой логирования.

## Рекомендации по производительности и масштабированию (5000+ id)
Когда требуется обработать тысячи ID в одном запросе, необходимо учитывать память, транзакции, блокировки и время отклика.

Чтобы масштабировать обработку большого количества ID, можно использовать следующую архитектуру:

1. API принимает bulk-запрос и сразу возвращает идентификатор задачи. Состав таких задач сохраняется в таблице `jobs`.
2. Разбить список ID на чанки и поместить задачи в очередь через Kafka/RabbitMQ или в локальную очередь ThreadPoolTaskExecutor.
3. Набор worker-потребителей читает чанки и выполняет транзакционные обновления по чанку. То есть, для каждого ID выполняется атомарный UPDATE, проверяется количество обновленных строк и, при успехе, сохраняет запись утверждения.
4. Обновлять прогресс задачи в базе по завершению каждого чанка.
5. Добавить возможность повтора выполнения задач для конфликтов и очередь dead-letter для необработанных ID.

Такой подход позволит не удерживать большую транзакцию, уменьшить время блокировок и обеспечить устойчивость при пиковых нагрузках. 
Кроме того, использование job-системы позволит отслеживать прогресс и повторять неудачные части без необходимости повторного запуска всего процесса.

## Вынесение реестра утверждений в отдельную систему
Подобную задачу можно решить двумя способами, в зависимости от требований к производительности и сложности архитектуры:

### 1) Отдельная БД
То есть, используется второй источник данных для реестра утверждений. В этом случае следует учитывать, что транзакция будет разделена на две части: 
- основная транзакция для обновления статуса документа,
- отдельная транзакция для записи в реестр утверждений.

В таком случае можно добиться достаточно легкого горизонтального масштабирования реестра, а также возможности репликации и архивации данных. 
Однако, важно учитывать, что при разделении транзакций может возникать ситуация, когда статус документа обновляется, но запись в реестре утверждений не создается, что может привести к несогласованности данных. 
Поэтому необходимо реализовать механизм обработки ошибок и повторных попыток для обеспечения целостности данных.

### 2) Отдельный микросервис
Такой способ подразумевает создание отдельного микросервиса, который будет отвечать за реестр утверждений. Взаимодействие с ним можно организовать через брокеры сообщений или через REST API (ну или GRPC, смотря какие будут требования).

В этом случае, после успешной транзакции по документу, сервис обработки документов будет отправлять POST-запрос в сервис реестра для записи факта утверждения. 
Сервис реестра будет обрабатывать запросы и сохранять данные в своей собственной базе данных. 
А при возникновении ошибки запроса необходимо реализовать механизм повторных попыток или отправки в очередь сообщений для последующей обработки.

Такой подход позволяет полностью изолировать логику реестра утверждений от основной бизнес-логики обработки документов, что может упростить поддержку и масштабирование систем.
